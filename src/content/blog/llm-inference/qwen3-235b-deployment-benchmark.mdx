这是一份 Qwen3-235B 级别模型部署压测记录的公开整理版。原始记录包含具体表格数据；公开版只保留指标口径、配置维度和结果解读方法，不包含内部测试数值。

注意：不同硬件、驱动、模型权重格式、vLLM 版本和 workload 都会显著影响结论。这里讨论的是怎么比较，而不是给出通用性能排名。

## 指标说明

| 指标 | 含义 |
| --- | --- |
| Total Token 吞吐 | 输入 token + 输出 token 的整体处理吞吐 |
| Output Token 吞吐 | 只统计输出 token 的吞吐 |
| TTFT | Time To First Token，首 token 延迟 |
| TPOT | Time Per Output Token，单输出 token 平均耗时 |

对于 235B 级别模型，TTFT 通常被长输入 prefill 主导，TPOT 更接近 decode 阶段体验。

## 双节点配置怎么读

双节点方案通常会组合 DP、TP、EP 和 DCP。它的目标不是单纯把模型“切开”，而是在吞吐、首 token 延迟、decode 稳定性和跨节点通信成本之间找平衡。

公开压测表里，我会至少记录：

- 并发档位。
- 输入长度区间。
- 输出长度上限。
- Total token throughput。
- Output token throughput。
- TTFT。
- TPOT。
- 显存占用和 KV Cache 占用。

双节点方案最需要警惕的是跨节点通信。总吞吐可能看起来更高，但如果 TTFT 和 TPOT 随并发快速恶化，交互式业务体验会被拖垮。

## 长输入场景怎么测

235B 级别模型常常被用在长上下文任务里。长输入压测要分开看两件事：

- prefill 阶段是否成为绝对瓶颈。
- decode 阶段是否被并行策略或通信成本拖慢。

如果只看 Total token throughput，长输入会把 prefill token 算进去，容易让结果看起来很好。交互式服务更应该同时看 TTFT、TPOT 和 P99。

## 单节点对比怎么做

单节点对比通常包括三类配置：

- TP / EP baseline。
- 加入 PP 的配置。
- 加入 DCP 的配置。

baseline 的价值是给出最朴素的延迟地面。PP 可能改善显存压力，但会引入 pipeline bubble。DCP 可能改善长上下文处理，但也可能改变 decode 阶段的稳定性。

比较时不要只问“谁的吞吐最高”，而应该问：

- 吞吐提升是否来自更高并发，还是单请求效率真的变好。
- TTFT 是否还能接受。
- TPOT 是否会影响流式输出体验。
- 显存余量是否足够支撑线上波动。

## 结果观察

这类压测最容易出现三个误判。

第一，长输入场景下不能只追吞吐。更高并发可能带来更高总 token 处理量，但 TTFT 和 TPOT 可能已经不可接受。

第二，PP、EP、DCP 的收益不是线性的。某个组合在短输入下表现好，不代表在长输入或高并发下也稳。

第三，baseline 不能省。没有 baseline，就很难判断复杂并行策略带来的收益是否覆盖了通信、调度和 pipeline 的额外成本。

## 下一步怎么测

后续建议补三类数据：

1. P50 / P95 / P99，而不是只看平均 TTFT 和 TPOT。
2. 显存占用、KV Cache 使用量、expert load balance。
3. 相同并发下的固定请求序列，避免 workload 随机性影响结论。
4. 分别记录 prefill-heavy、decode-heavy 和混合 workload。

如果结合 KV Cache 复用测试，还应该把长输入 prefix 的 hit rate 和 TTFT 下降幅度放在同一张表里。否则很难判断优化是来自并行策略，还是来自缓存命中。

## 小结

Qwen3-235B 这种规模的模型，部署策略不是简单地“并发越高越好”。双节点配置能支撑更复杂的并行策略，但延迟也更容易被通信和调度放大。真正上线前，需要结合业务对 TTFT、TPOT 和总吞吐的权重来选点。
