KV Cache 复用的收益不应该只看“有没有命中”。更关键的是：在哪类 workload 下命中、命中后节省了多少 prefill、传输和 offload 又额外引入了多少成本。

这篇整理两类更容易复现的公开测试场景：多轮对话和长文本问答。具体压测规模、容量配置和内部环境信息已省略，只保留方法论。

## 场景一：多轮对话

测试设定可以抽象成下面几个维度：

| 项目 | 设置 |
| --- | --- |
| 用户规模 | 选择一个能稳定压出排队现象、但不会直接打爆服务的并发档位 |
| 每轮 prompt | 使用固定长度区间，保证不同轮次之间可比较 |
| 每轮回答 | 固定输出上限，减少 decode 侧波动 |
| 轮数 | 覆盖 cold、warm 和 cache hit 三类阶段 |
| 请求分发 | 让同一用户的请求有机会落到不同实例 |
| 缓存层 | 对比无共享缓存、本地缓存、跨实例共享缓存 |

这个场景模拟的是“同一个用户持续对话”的缓存复用。前几轮会逐步积累上下文，后续请求如果能复用历史 prefix，就有机会减少重复 prefill。

## 为什么多轮对话适合测 KV Cache

多轮对话的优势是 prefix 关系明确：

- 同一用户的后续请求天然包含前文。
- prompt 增长是渐进的，不是一次性超长输入。
- 可以分开观察 cold round、warm round 和 cache hit round。
- 请求在两个实例之间随机分发，可以验证跨实例 KV 共享是否生效。

如果只在单实例上测，多轮对话的收益可能被本地缓存掩盖。随机打到两个实例后，才能更清楚地看出 LMCache 这类共享缓存层的价值。

## 场景二：长文本问答

测试设定可以抽象成下面几个维度：

| 项目 | 设置 |
| --- | --- |
| 输入结构 | 长文档 prefix + 短问题 |
| 输出长度 | 固定输出上限，便于比较 TPOT |
| 请求分发 | 请求按固定节奏进入多个实例 |
| 缓存容量 | 使用足够容纳热点 prefix 的容量档位 |
| 对比组 | 无缓存、单实例缓存、跨实例共享缓存 |

这个场景模拟的是 RAG、文档问答、长报告分析这类 workload。它的特点是 prefix 很长，回答相对短。

## 为什么长文本问答更考验系统

长文本问答对 KV Cache 更敏感，也更容易暴露问题：

- prefill 成本高，理论上复用收益更大。
- 输入长度长，KV 占用更高，缓存容量压力更明显。
- 如果跨实例传输成本过高，收益会被 RDMA/网络/序列化成本吃掉。
- 如果 cache chunk 粒度不合适，会出现 partial hit 收益不稳定。

因此长文本问答不能只看平均吞吐。还需要看 TTFT、P99、hit rate、partial hit 比例，以及不同实例之间的数据迁移成本。

## 结果解读口径

整理成 benchmark 时，我会把结果表统一成下面几个维度：

| 指标 | 解释 |
| --- | --- |
| TTFT | 首 token 延迟，最能反映 prefill 是否被有效节省 |
| TPOT | 每输出 token 耗时，反映 decode 阶段是否被缓存系统拖慢 |
| Total token throughput | 输入 + 输出整体吞吐，适合观察系统总处理能力 |
| Output token throughput | 纯输出吞吐，适合对比 decode 侧能力 |
| hit rate | 请求级、token 级和 chunk 级最好分开统计 |
| offload latency | CPU / 本地磁盘 / 远端缓存传输的额外耗时 |

## 实验建议

第一版实验可以这样拆：

1. 单实例，无 LMCache，作为 baseline。
2. 双实例，打开 LMCache，但先不启用远端 offload。
3. 双实例，启用 CPU offload。
4. 双实例，启用远端共享 KV。
5. 固定 workload，逐步增加用户数和输入长度。

每一步只改一个变量，才能判断收益来自 KV 复用、实例间共享，还是单纯来自并发变化。

## 小结

多轮对话和长文本问答是 KV Cache 最应该优先验证的两个场景：前者验证“历史上下文复用”，后者验证“长 prefix 复用”。一个看会话连续性，一个看大块 prefix 成本。把这两个场景测清楚，后续再扩展到 RAG、问学、代码仓库问答或 Agent 记忆，结论会更扎实。
