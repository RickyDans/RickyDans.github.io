在 Kubernetes 上跑推理服务，第一步不是直接部署模型，而是确认集群能稳定识别 GPU、Helm 可用、GPU Operator 安装策略清楚。下面是一份最小检查清单。

## 节点状态

先确认节点能被控制面看到：

```bash
sudo kubectl --kubeconfig=/etc/kubernetes/admin.conf get nodes
```

再查看是否已有 GPU 相关组件：

```bash
sudo helm --kubeconfig=/etc/kubernetes/admin.conf list -A | grep -Ei "gpu|nvidia|operator"
```

如果这里已经有 device plugin、GPU Operator 或自定义安装的驱动组件，需要先弄清楚现状，避免重复安装后互相覆盖。

## 安装 Helm

一份可复用的 Helm 安装脚本大致如下：

```bash
HELM_BUILDKITE_APT_KEY_ID="DDF78C3E6EBB2D2CC223C95C62BA89D07698DBC6"

sudo apt-get install curl gpg apt-transport-https --yes
curl -fsSL https://packages.buildkite.com/helm-linux/helm-debian/gpgkey > "${TMPDIR:-/tmp}/helm.gpg"
gpg --show-keys --with-fingerprint --keyid-format LONG "${TMPDIR:-/tmp}/helm.gpg"

sudo mkdir -p /usr/share/keyrings
sudo gpg --dearmor -o /usr/share/keyrings/helm.gpg "${TMPDIR:-/tmp}/helm.gpg"

echo "deb [signed-by=/usr/share/keyrings/helm.gpg] https://packages.buildkite.com/helm-linux/helm-debian/any/ any main" | \
  sudo tee /etc/apt/sources.list.d/helm-stable-debian.list

sudo apt-get update
sudo apt-get install helm
```

生产环境里建议把 key fingerprint 校验写进脚本，而不是人工看一眼就继续执行。

## 安装 NVIDIA GPU Operator

添加 NVIDIA Helm repo：

```bash
helm repo add nvidia https://helm.ngc.nvidia.com/nvidia --force-update
helm repo update nvidia
```

如果节点上已经手动安装了驱动和 NVIDIA Container Toolkit，可以关闭 operator 的 driver/toolkit 管理：

```bash
helm install gpu-operator nvidia/gpu-operator \
  --namespace gpu-operator \
  --create-namespace \
  --set driver.enabled=false \
  --set toolkit.enabled=false
```

这组参数的含义是：GPU Operator 管理 Kubernetes 侧组件，但不接管宿主机驱动和 toolkit。对于已有 GPU 节点，这是比较稳的第一步。

## 安装后检查

安装后建议至少确认：

- `gpu-operator` namespace 下的 pod 是否全部 ready。
- 节点上是否暴露 `nvidia.com/gpu` 资源。
- 容器内是否能执行 `nvidia-smi`。
- RuntimeClass 或默认 runtime 是否符合预期。
- 现有驱动版本是否和 CUDA / PyTorch / vLLM 镜像兼容。

## Dynamo 入口

如果目标是 NVIDIA Dynamo，可以在 GPU Operator 稳定后再进入 Dynamo 的 Kubernetes 部署文档，继续处理 CRD、服务拆分、调度和观测配置。

一个实用原则是：先让普通 GPU workload 在集群里跑通，再上 Dynamo。这样后续遇到问题时，能判断是 Kubernetes/GPU 基础设施问题，还是 Dynamo 自身部署问题。

## 小结

GPU 推理集群的第一阶段应该保持克制：节点可见、Helm 可用、GPU Operator 状态稳定、GPU 资源能被 pod 消费。这个基础层稳定后，Dynamo、vLLM、LMCache 或其他推理系统才有可靠的调试地面。
